# Research — Daily Rolling Grocery-List Refresh (`impl/nextjs`)

Phase 0 output. All Technical Context unknowns resolved; decisions numbered for traceability from tasks. Grounded in the shipped 006/007 code: `src/server/lib/grocery-list-generator.ts`, `src/server/controllers/grocery-lists.ts`, `src/server/models/grocery-list.ts`, `src/server/lib/expiration.ts`, `src/lib/date-utils.ts`.

## D1 — Rolling list is computed presentation over the existing week-keyed document

**Decision**: Keep the `GroceryList` collection exactly as it is — one document per `(userId, weekStart)`, unique-indexed. "Rolling rest-of-week" (FR-RG-008) is realised by making the document's **content** a function of the viewing day, not by introducing a new document type or a "rolling list" entity. The client's default view stays the current week (`getWeekStart(0)` via `MealPlanContext.currentWeekStart`), so today→end-of-week is simply the in-scope subset of the current week's document. A fully-past week's document naturally recomputes to empty generated needs (FR-RG-009); it is never surfaced as browsable history.
**Rationale**: Preserves the spec-007 tick/un-tick/checkout flows, which operate on real stored rows with stable `_id`s (FR-RG-011). Minimal blast radius: no migration, no new index, no new context. Matches the clarified assumption that "how the rolling view is realised over week-keyed storage is a plan decision."
**Alternatives considered**: A new per-day or per-scope document — rejected: duplicates identity, breaks 007's row-anchored receipts, needs migration. A read-time-only projection that never persists — rejected under D4 (generated rows need stable ids for ticking).

## D2 — One date-scoped generation path shared by lazy GET and force-generate

**Decision**: Extract a controller helper `reconcileRollingList(existing, mealPlan, inventory, asOf)` in a new pure lib `src/server/lib/rolling-grocery.ts`, and call it from **both** `getGroceryList` (recompute-on-view) and `regenerateGroceryList` (force). `generateGroceryList(mealPlan, inventory, asOf)` gains an `asOf: Date` parameter and filters `planned` entries to those dated today-or-later before its existing netting/servings logic runs (FR-RG-001/003). The reconcile helper then merges those fresh needs into the stored document.
**Rationale**: FR-RG-002 requires manual and automatic refresh to produce the same list; one code path guarantees it. `generateGroceryList` already isolates the netting engine (FR-MC-016..018 / FR-026 fallback), so a scope parameter is the smallest change that re-scopes everything — including the servings fallback, whose count now sums only in-scope meals (FR-RG-003, EC "mixed-source line partially in the past").
**Alternatives considered**: Duplicate the date filter in GET and generate — rejected (drift risk, FR-RG-002 violation). Filter inside the Mongo query — rejected: entry status is derived (`entryStatus()` treats absent status as cooked) and netting needs the full inventory in memory anyway.

## D3 — "Today" is derived server-side; the cutoff maps the user's local day onto the entries' UTC-midnight axis

**Decision**: `startOfTodayCutoff()` in `rolling-grocery.ts` reads the **server clock** (`new Date()`), takes the user's **local** calendar day (`getFullYear/Month/Date`), and returns that day as a UTC-midnight instant: `new Date(Date.UTC(y, m, d))`. An entry is in scope iff `entry.date.getTime() >= cutoff.getTime()`. "Today" is never accepted from the client.
**Rationale**: (1) *Trust/correctness* — single-user LAN, the server is authoritative; a client-supplied "today" would be spoofable and pointless. The client already supplies only `weekStart` (which week), a navigational choice; scope is a server concern. (2) *Timezone correctness (FR-RG-010)* — meal-plan entry dates are authored at **UTC** midnight (`getWeekDays` uses `Date.UTC`), but the user's real "today" is their **local** calendar day, and the LAN host runs in the user's timezone. Taking local Y/M/D and re-projecting onto the UTC-midnight axis compares *calendar day to calendar day* and avoids the off-by-one that a naive local-vs-UTC compare would cause near midnight (e.g. AEST evening = next UTC day). (3) *Consistency* — mirrors the midnight-cutoff spirit of `lib/expiration.ts` (`setHours(0,0,0,0)`), the convention FR-RG-010 points at. A today-dated entry (`date == cutoff`) stays in scope all day (`>=`), matching "counts all day."
**Alternatives considered**: Local-midnight instant compared directly to UTC-dated entries (`expiration.ts` style) — rejected: off-by-one for any non-UTC timezone because the two axes differ. Pure UTC "today" (`Date.UTC(nowUTC)`) — rejected: in AEST evening it still reads yesterday's UTC date and would keep a passed meal in scope. Client-supplied ISO "today" — rejected on trust grounds. *This local-vs-UTC seam is the one tension the tasks phase must lock down with an explicit boundary test (`vi.setSystemTime` at 23:59 / 00:01 either side of the local day).*

## D4 — Generated rows are reconciled in place by `ingredientName`, not wiped-and-recreated

**Decision**: On recompute, partition the stored `items` into **sticky** (manual, or purchased/receipted) and **replaceable** (generated + unpurchased + no receipt). Replaceable rows are diffed against the freshly generated needs keyed by `ingredientName`: where the name survives, keep the existing `_id` and overwrite `quantity`/`unit`/`sourceMealNames` (FR-RG-007); where the recomputed need is gone or zero, drop the row (FR-RG-006); where a need is new, insert a fresh row. Sticky rows are carried through verbatim (subject to the D5 shed).
**Rationale**: FR-RG-011 requires that a mid-view tick still targets a live row. Preserving `_id` for surviving generated lines keeps the client's in-flight PATCH valid across a refresh; only genuinely-removed lines change identity. This turns FR-RG-006 (removal) and FR-RG-007 (requantify) into a natural diff rather than a churn-everything rewrite.
**Alternatives considered**: Blow away all generated rows and re-insert each recompute — rejected: `_id` churn 404s in-flight ticks and defeats the 007 row anchor. Match on `displayName` — rejected: `ingredientName` is the normalized canonical key the generator already produces (`normalizeIngredientName`).

## D5 — Day-anchoring: `addedOn` / `purchasedOn` stamps; shed is a write-time prune

**Decision**: Add two optional `Date` fields to the item subdoc — `addedOn` (set when a manual item is created in `addGroceryItem`) and `purchasedOn` (set when a line is ticked purchased in `purchaseGroceryItem`). A sticky row's **anchor day** is `purchasedOn ?? addedOn`. During recompute, a sticky row is preserved iff its anchor day ≥ `startOfTodayCutoff()`; otherwise it is **pruned** (shed) from the document — a write-time prune, so the stored document converges to the rolling view every view (FR-RG-004/005). Shedding drops the row **and its `purchaseReceipt`**; the referenced `InventoryItem` is never touched (FR-RG-005/011). Legacy sticky rows lacking any anchor are **lazily backfilled** to the recompute day on first 008 recompute (so a pre-deploy purchase survives its first rolled-over day rather than vanishing), then shed normally at the next rollover.
**Rationale**: The anchor is the minimal state needed to answer "which day does this row belong to." Write-time prune (vs a read-only filter) keeps the persisted document honest so 007's PATCH/complete keep operating on exactly what the user sees, and keeps the un-tick 409/404 semantics coherent: within the day the row (with receipt) exists → un-tick reverses per receipt (FR-RG-011); after shed the row is gone → un-tick 404s and reversal is not offered (FR-RG-005). Receipt retention past shed is explicitly optional in the spec (no user-facing history required), so dropping it with the row is simplest and the inventory addition stays final. Lazy backfill avoids a destructive migration and a surprise mass-shed on deploy.
**Alternatives considered**: Read-time filter that leaves shed rows persisted — rejected: the stored doc would disagree with the view, and a stale receipted row could still be un-ticked out of its window. A separate `sheddedAt` tombstone / receipt archive — rejected: no history requirement, needless storage. Anchor stored as a day-string — rejected: `Date` reuses the existing midnight-cutoff comparison and Mongoose date handling.

## D6 — Netting and the servings fallback are inherited unchanged under the reduced scope

**Decision**: Within the date-reduced meal set, quantity netting follows the existing rules verbatim: sum grounded needs per canonical name, net off non-expired owned stock via `notExpiredQuery()` (FR-MC-016..018), omit fully-covered ingredients (`shortfall <= 0`), and fall back to the servings count where unit families cannot be reconciled (FR-MC-017 / FR-026) — with the servings count now summing only in-scope meals (FR-RG-003). No change to `unit-normalizer`, `ingredient-matcher`, or `ingredient-categorizer`.
**Rationale**: FR-RG-003 mandates "existing rules unchanged" inside the smaller scope. Passing pre-filtered entries into the untouched engine achieves exactly that, including the mixed-family servings fallback and the "mixed-source line partially in the past" edge case (the line shrinks to the remaining meals' shortfall; servings likewise count only today-onwards). Shed purchases already sit in inventory, so their stock nets off future needs and nothing is re-listed (FR-RG-011).
**Alternatives considered**: Re-implement netting with a date filter — rejected: duplication of FR-MC-016..018 logic. Filter needs *after* generation — rejected: would mis-net stock against out-of-scope demand.

## D7 — Wrong-state and boundary responses reuse the 007 contract

**Decision**: The 007 `PATCH .../items/:itemId` and `POST .../complete` request/response surfaces are **unchanged**. The only new observable consequences are date-scoped: (a) a generated row that sheds/zeroes is simply absent from the next GET (no error); (b) an un-tick of a row that has already **shed** returns **404** (row gone), not the 409 that a same-day receipt-less/duplicate un-tick returns — the client treats both as "cannot reverse, refetch." Recompute never returns an error for scope changes.
**Rationale**: Keeps 008 a pure superset of 007 at the wire level (SC — "check-off flows operate unchanged"), so no client contract break. The 404-after-shed is the natural result of the write-time prune (D5) and needs only a one-line contract note.
**Alternatives considered**: A dedicated 409 "shed" reason — rejected: over-specifies a state the user cannot act on; 404 is already the "row not found" path the client handles.
