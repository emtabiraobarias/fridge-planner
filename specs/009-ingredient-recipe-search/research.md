# Research — Ingredient-Driven Recipe Search + Manual-Only Recommendations (`impl/nextjs`)

Phase 0 output. All Technical Context unknowns resolved; decisions numbered for traceability from tasks. Grounded in the shipped 001/006/007 code: `src/server/controllers/recommendations.ts`, `src/server/services/recommendations-cache.ts`, `src/server/services/meal-recommender.ts`, `src/server/lib/ingredient-grounding.ts`, `src/server/lib/purchase-inventory.ts`, `src/server/controllers/inventory.ts`, `src/components/recommendations/RecommendationsPanel.tsx`, `src/components/calendar/SuggestionsRail.tsx`, `app/providers.tsx`, `src/context/ToastContext.tsx`.

## D1 — Scope is transmitted as an optional `ingredientItemIds: string[]` of inventory `_id`s

**Decision**: `POST /api/v1/recommendations` gains one optional body field, `ingredientItemIds?: string[]` — the `_id`s of the selected inventory items — validated in the route with Zod (array of 1–64-char strings, capped at the grounding `MAX_INGREDIENTS` = 20; malformed/absent → treated as no selection). The route passes it to `getRecommendations(userId, ingredientItemIds?)`. The controller loads the user's live non-expired inventory exactly as today (`InventoryItem.find({ userId, ...notExpiredQuery() })`, `recommendations.ts:23`) and **intersects by id**: `ids?.length ? activeItems.filter(i => ids.includes(String(i._id))) : activeItems`. That subset feeds the unchanged pipeline; spec 006 grounding then re-validates against exactly the subset because `groundMeals(userId, meals, scopedItems)` already takes the inventory array as its third argument (`ingredient-grounding.ts:173`).
**Rationale**: The selection is of **real inventory items** (FR-IR-005 inventory-only), so ids are the unambiguous, spoof-resistant key — they line up with spec 006's tier-1 direct-id grounding (`ingredient-grounding.ts:114`), the server already has `_id` per item, and the client already renders items keyed by `_id`. Validating server-side and intersecting with the live `notExpiredQuery()` load means a stale/deleted/edited/expired selected id is silently dropped (FR-IR-008, spec EC "selected item deleted before the request completes"). The field is optional so a whole-inventory request sends `{}` exactly as today (`services/inventory.ts:119`) — zero contract break.
**Alternatives considered**: `ingredientNames: string[]` — rejected: names collide and are ambiguous, and would force a name→item re-match the id already gives us for free. A separate `POST /recommendations/scoped` endpoint — rejected: duplicates the fallback ladder, rate limit, and grounding; the spec's whole theme is one reused pipeline with a changed input. Sending the full ingredient objects from the client — rejected on trust grounds (server must re-load and re-validate anyway).

## D2 — Per-selection caching falls out of the existing `buildCacheKey` for free

**Decision**: Do **not** touch `recommendations-cache.ts`. `buildCacheKey(userId, ingredients)` already derives the key from the sorted set of `name|quantity|unit|expiresAt` tuples (`recommendations-cache.ts:12-21`). Because a scoped request maps only the selected subset into `ingredients`, its key is naturally distinct from the whole-inventory key and from other selections — so repeating an identical selection within the 15-min TTL is a cache hit (`getCached`, FR-IR-011/SC-IR-005) with no new cache mechanism. The scoped-subset ingredient set = the selected items' `{name, quantity, unit, expiresAt}` tuples (the same shape the whole-inventory path builds, just fewer rows).
**Rationale**: The cache is keyed on *what the agent will be asked about*, which is exactly the scoped ingredient set — so correctness (same ingredients → same grounded meals) and per-selection isolation are both inherent. A whole-inventory request and a full-selection request key identically only if the tuple sets are identical, which the spec explicitly accepts (Assumptions: "a whole-inventory request and a full-selection request may key differently and that is acceptable"). `invalidateUser(userId)` still clears every scoped and whole key on any inventory mutation (`recommendations-cache.ts:42-47`), so no scoped entry outlives the inventory it described.
**Alternatives considered**: Add the id set to the cache key — rejected: two selections with identical `{name,qty,unit,expiresAt}` tuples *should* share a result (they ground identically), and ids would over-fragment the cache and split legitimately-shareable hits. A separate scoped cache map — rejected: needless; the one keyed map already discriminates.

## D3 — One contextual action: the client omits the scope field when the selection is empty; the server also guards

**Decision**: A single "Get Recommendations" / "Find recipes with selected" action routes both modes (FR-IR-007). Client-side, the service call includes `ingredientItemIds` **only** when a selection is active and non-empty; an empty selection sends `{}` → whole inventory (FR-IR-004/010). The server additionally guards: if the provided ids intersect live non-expired inventory to **zero** items (empty, or only-expired selection), it falls back to the whole non-expired set rather than sending the agent an empty ingredient list (FR-IR-010; a zero-length scoped request would otherwise degrade to EC-01 popular-recipes, which is wrong for a user who has stock).
**Rationale**: Client-side omission keeps the "one contextual action" ergonomics (the button's *label* changes with selection presence, but there is one code path, one endpoint). The server guard is defence-in-depth for the spec's edge cases (all-expired selection, selection emptied server-side between tap and request) so the outcome is deterministic regardless of what the client sent.
**Alternatives considered**: Always send `ingredientItemIds` (empty array = whole inventory) — rejected: an empty array is ambiguous with "the user genuinely selected nothing then hit scoped," and muddies the whole-inventory equivalence FR-IR-004 wants to keep byte-identical. Two separate buttons/endpoints for the two modes — rejected by FR-IR-007 ("no separate always-visible second button").

## D4 — Removing auto-load without breaking session persistence

**Decision**: The auto-load to remove is the prefetch `useEffect` in `RecommendationsPanel.tsx:38-43` — it fires `handleFetch()` when inventory first transitions from empty to non-empty, spending an agent call on the Kitchen/home screen (`/`, the login landing) with no user action. Delete that effect; keep the existing "Get Recommendations" button as the only trigger and show a calm empty state before the first request (FR-IR-001/002). Session persistence (FR-IR-003) requires **no new state**: `RecommendationsProvider` is mounted app-level in `app/providers.tsx:23`, above the router, so `meals`/`cachedAt`/`state` survive navigation between `/`, `/calendar`, `/grocery` for the session; and `handleFetch` already returns early when `meals.length > 0` within the client TTL (`RecommendationsPanel.tsx:54-55`), so returning to the screen shows prior results with no new request.
**Rationale**: The "already fetched this session" state the spec relies on is the existing in-memory `RecommendationsContext` (spec Assumptions: "a session-scoped in-memory result (existing behaviour) is what 'already fetched this session' relies on — no new persistence"). The only defect is the *automatic* trigger; the persistence and the manual button are already correct. The calendar `SuggestionsRail` is **already** manual (a "Get suggestions" button gated on `meals.length === 0`, `SuggestionsRail.tsx:35-44`, no auto-fetch) — it needs no removal, only the IR2 chip entry point and any empty-state copy.
**Alternatives considered**: Persist results in `sessionStorage` — rejected: the app-level context already provides session-lifetime persistence; storage would duplicate it and risk serving results across inventory mutations that `invalidateUser` should have dropped. Move recommendations to a route-local state — rejected: would *lose* cross-screen persistence FR-IR-003 needs.

## D5 — The two entry points converge at the service/endpoint, not a shared selection context

**Decision**: Keep the ingredient selection **transient and local to each surface** — Kitchen select-mode state lives in `InventoryPage`/`InventoryList` (checkbox toggles → a `Set<string>` of item ids); suggestions-rail chip state lives in `SuggestionsRail` (chip toggles → the same id-set shape). Both build an `ingredientItemIds` array and call the **one** `fetchRecommendations(ids)` service, whose result flows into the **one** app-level `RecommendationsContext`. No new `SelectionContext`.
**Rationale**: The user is on one screen at a time and the selection is explicitly "not persisted beyond the request/session" (spec Key Entities / Assumptions). What must be *shared* is the **result**, and that already is (app-level `RecommendationsProvider`). Convergence "on one scoped request" (FR-IR-006/SC-IR-004) is satisfied at the service+endpoint layer — both surfaces produce byte-identical output for the same id set because they hit the same controller path. Adding a cross-screen selection atom would violate the "transient selection" model and add a context the constitution says to avoid.
**Alternatives considered**: An app-level `SelectionProvider` shared by both surfaces — rejected: implies cross-screen selection persistence the spec forbids, and adds a context for no behavioural gain. Duplicating the fetch logic in each surface — rejected: both must call the identical service so results are provably equivalent (SC-IR-004); the shared function is the single source of truth.

## D6 — EC-03 quick-add merge reuses the spec-007 matcher server-side, gated by an opt-in flag

**Decision**: Extract the spec-007 matching predicates already in `purchase-inventory.ts` — `sameIngredient` (`normalizeIngredientName` equality, `:29`), `canMergeUnits` (`canSubtract || isSameResolvedUnit`, `:37`), `convertQuantity` (`:41`), and the non-expired candidate load `sameNameCandidates` (`notExpiredQuery()`, `:49`) — into a new pure lib `src/server/lib/inventory-merge.ts` exposing `findMergeTarget(userId, name, unit)` and `mergeInto(target, quantity, unit)`; refactor `applyPurchase` to consume it (behaviour-preserving, keeps the 007 suite green). `createInventory` gains an **opt-in** `mergeDuplicates?: boolean` in its Zod schema, set only by the quick-add path: when true and `findMergeTarget` returns a same-name, non-expired, compatible-unit item, increment that item's quantity (converted) instead of inserting, and return a merge indicator `{ merged: true, mergedItemId, addedQuantity }`; expired or unit-incompatible same-name items are not targets → create a new row, `{ merged: false }` (FR-IR-012, explicitly "consistent with spec 007 FR-GC-005").
**Rationale**: Merge eligibility (canonical-name match, unit-family compatibility, non-expired gating) is server-only logic that already exists and is battle-tested in the grocery check-off flow; reusing it is the spec's central "reuse over rebuild" mandate and the constitution's "no duplicate date/logic." Server-side is the correct layer — the client cannot see the full authoritative inventory or run `unit-normalizer`. The opt-in flag scopes the new behaviour to quick-add (FR-IR-012 is quick-add-specific), so deliberate "create a new item" callers (e.g. a future explicit add form) keep today's semantics with no surprise merges. The additive indicator lets the client offer Undo without a persisted receipt.
**Alternatives considered**: Merge on **every** `POST /inventory` — rejected: would silently merge deliberate creates, surprising and out of FR-IR-012's scope. Do the match client-side before POST — rejected: duplicates server-only matching, races against live inventory, and can't reach `unit-normalizer`. A new `POST /inventory/quick-add` endpoint — rejected: `createInventory` + a flag is smaller and keeps one create path.

## D7 — Undo is a transient client-held reversal mirroring `reversePurchase`, not a persisted receipt

**Decision**: On a merge, the client keeps a transient snapshot `{ mergedItemId, addedQuantity }` (in the add-flow local state, surfaced through an action-capable toast). Tapping Undo issues a normal inventory edit that **subtracts `addedQuantity` from the item's current quantity, clamped at ≥0** (and deletes the row if the merge had created it and the delta is the whole quantity) — exactly the shape of spec 007's `reversePurchase` (`purchase-inventory.ts:160-171`). No server-side merge receipt is stored (FR-IR-013; spec Assumptions: "Undo is a client-side reversal … transient, no persisted receipt required"). To carry the Undo control, `ToastContext` (`showToast(message)` today, `ToastContext.tsx:7`) is extended with an optional `action?: { label; onAction }`; existing message-only calls are unchanged.
**Rationale**: Subtract-delta-and-clamp resolves the spec's flagged "Undo after further edits" edge precisely — if the user edited the merged item between merge and Undo, subtracting the recorded delta preserves their edit minus the merge and can never produce a negative or phantom quantity ("define precedence so Undo never produces a negative or phantom quantity"). A hard reset to a captured pre-merge absolute would instead clobber an intervening edit, which is worse. Reusing the `reversePurchase` shape keeps the reversal semantics identical to the grocery path the spec says to mirror, without the persistence that path needs for its cross-request receipt.
**Alternatives considered**: Reset the item to a captured absolute pre-merge quantity — rejected: clobbers intervening edits and can go phantom. A persisted transient receipt on the inventory item — rejected: the spec explicitly wants no persisted receipt for a quick-add merge, and it would need a schema field for a two-tap-lifetime affordance. A dedicated `POST /inventory/:id/undo-merge` — rejected: the existing `PUT`/`DELETE` inventory endpoints already express the reversal.

## D8 — Expiry prioritisation within a selection is emergent; no server change

**Decision**: FR-IR-009 (a sooner-to-expire selected item is favoured within the selection) needs **no** server logic beyond D1's subset filter. The meal-recommender already instructs the agent to "Prioritise ingredients expiring soonest to minimise food waste" and passes each item's `expiresAt` (`meal-recommender.ts:32-33,47-48`). Because the agent only ever sees the items the controller sends, scoping the input to the selection makes expiry prioritisation apply *within* the selection automatically.
**Rationale**: The prioritisation lives in the agent prompt over whatever ingredient set it receives; reducing that set to the selection is the entire mechanism. Adding a server-side re-sort would duplicate the agent's job and risk diverging from the whole-inventory behaviour FR-IR-004 keeps identical.
**Alternatives considered**: Server-side re-rank of returned meals by selected-item expiry — rejected: redundant with the agent, and the grounded result already reflects the prompt's prioritisation.

## D9 — FR-037 lazy links, the rate limit, and EC-01/EC-08 fallbacks are inherited unchanged

**Decision**: `POST /recommendations/verify-links` and its controller `verifyRecipeLinks` are **untouched** — they key on meal names and are independent of input scope, so the FR-037 lazy phase runs identically on scoped results (spec EC "Recipe-link verification on scoped results"). The `rateLimit('recommendations:${userId}', 10, 60_000)` tier (`recommendations/route.ts:16`) is shared by scoped and whole-inventory requests (FR-IR-011). EC-01 (empty inventory → `POPULAR_RECIPES`) and EC-08 (agent down → stale cache else popular) remain exactly as coded (`recommendations.ts:27-29,52-59`); a scoped request with a non-empty selection simply enters the same ladder with fewer ingredients, and an empty/all-expired selection is routed to whole inventory first (D3) so EC-01 only triggers on genuinely empty inventory.
**Rationale**: These surfaces have no dependency on which ingredients were selected; changing them would be scope creep against the spec's "reuse unchanged" mandate.
**Alternatives considered**: A separate rate-limit budget for scoped requests — rejected: FR-IR-011 mandates the shared budget. Skipping FR-037 for scoped results — rejected: the spec requires it to run unchanged.
